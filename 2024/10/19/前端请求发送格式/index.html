常见请求类型:

前端发送请求的常见格式主要有以下几种：

1. GET 请求

  • 用于从服务器获取数据，没有请求体，数据通过 URL 参数传递。

2. POST 请求

  • 用于向服务器发送数据，数据一般在请求体中。

3. PUT 请求

  • 用于更新资源，数据通过请求体发送。

4. DELETE 请求

  • 用于删除服务器上的资源，通常没有请求体。
  • 示例：

5. PATCH 请求

  • 用于部分更新资源，与 PUT 类似，但只更新特定字段。
  • 示例：

6. 常见的请求头

  • Content-Type: 用来说明请求体的数据格式，如 application/json 表示请求体是 JSON 格式。
  • Authorization: 用来传递认证信息，比如 Bearer Token，用于身份验证。

7. 常见工具库

  • Axios: 是一个流行的 HTTP 客户端，基于 Promise，可以用于浏览器和 Node.js。其语法更简洁，自动处理 JSON 格式数据。

这些是前端发送请求的常见格式，取决于具体的需求，选择合适的请求方法和配置

常见的几种请求体数据格式：

1. JSON (JavaScript Object Notation)

  • 常用于现代 Web 开发，具有轻量级和易于阅读的特点。
  • 格式: {"key": "value"}
  • 适用场景: 大部分 RESTful API 使用 JSON 格式传输数据。
  • 示例：

2. XML (eXtensible Markup Language)

  • 类似 HTML 的标记语言，常用于老旧系统或特定领域的数据传输。
  • 格式: <root><key>value</key></root>
  • 适用场景: 早期的 Web 服务，尤其是基于 SOAP 的系统。
  • 示例：

3. Form Data (application/x-www-form-urlencoded)

  • 常用于 HTML 表单提交，数据被编码成键值对，使用 URL 编码。
  • 格式: key1=value1&key2=value2
  • 适用场景: 表单提交，例如登录、注册或搜索。
  • 示例：

4. Multipart Form Data

  • 用于上传文件或混合文本和二进制数据的表单提交。
  • 格式: 每个部分都包含其自己的内容类型和文件元数据。
  • 适用场景: 文件上传（例如图片、音频等）。
  • 示例：

5. Plain Text (text/plain)

  • 用于传输纯文本数据，不做任何格式转换。
  • 格式: 纯文本字符串
  • 适用场景: 简单的数据传输或日志文件传输。
  • 示例：

6. GraphQL

  • 一种查询语言，用于前端灵活获取后端 API 数据。
  • 格式: 查询结构与 JSON 类似，但直接书写查询语句。
  • 适用场景: GraphQL API 请求。
  • 示例：

7. Binary Data

  • 用于传输二进制数据，如图片、视频或音频文件。
  • 格式: 无需编码，直接传递二进制数据。
  • 适用场景: 多用于文件传输和处理。
  • 示例：图片上传等使用二进制文件流。

8. YAML (YAML Ain’t Markup Language)

  • 一种简洁易读的格式，适合用作配置文件，也可用于数据传输。
  • 格式: 类似于 Python 的缩进格式。
  • 适用场景: 主要用作配置，但某些 API 也支持此格式。

常见HTTP状态码

后端返回的错误代码通常遵循 HTTP 状态码 规范，这些状态码根据其范围可以分为不同的类别。以下是常见的错误类型和对应的状态码：

1. 4xx 客户端错误

  • 这类错误表示客户端发出的请求有问题，如请求格式不对、权限不足等。

  • 400 Bad Request:

    • 请求格式错误，服务器无法理解请求。
    • 例如：请求体中的 JSON 格式错误，或者缺少必填字段。

  • 401 Unauthorized:

    • 需要身份验证，未通过验证的用户无法访问资源。
    • 例如：没有提供有效的认证 Token。

  • 403 Forbidden:

    • 拒绝访问，即使用户已认证，也没有权限执行该操作。
    • 例如：用户尝试访问受限资源或没有权限的操作。

  • 404 Not Found:

    • 服务器无法找到请求的资源。
    • 例如：访问一个不存在的 API 路径或资源 ID。

  • 405 Method Not Allowed:

    • 请求的方法不被允许。
    • 例如：对某个资源发起 POST 请求，而服务器只接受 GET 请求。

  • 429 Too Many Requests:

    • 客户端请求过多，超出了服务器的限制（常见于速率限制）。

2. 5xx 服务器错误

  • 这类错误表示服务器在处理请求时发生了错误，可能是程序或数据库出错。
  • 500 Internal Server Error:
    • 服务器内部错误，通常是代码中的异常或未知错误。
    • 例如：后端代码逻辑错误、空指针异常、无法处理请求的数据等。
  • 502 Bad Gateway:
    • 网关或代理服务器从上游服务器接收到无效响应。
    • 例如：后端服务超时或崩溃，代理服务器无法获取有效响应。
  • 503 Service Unavailable:
    • 服务器不可用，通常是由于过载或维护。
    • 例如：服务被暂时关闭进行维护。
  • 504 Gateway Timeout:
    • 网关或代理服务器等待上游服务器的响应超时。
    • 例如：后端服务响应过慢或未响应。
  • 301
    • 请求地址已永久移动到新地址

3. 常见的应用层错误

  • Validation Errors (验证错误):
    • 例如表单字段不符合要求、数据格式不正确，通常在 API 返回时，HTTP 状态码可能是 400，并且在响应体中返回详细的验证错误信息。
  • Authentication/Authorization Errors (身份验证或权限错误):
    • 当用户未授权时，通常会返回 401 或 403，并带有消息解释如 “Token 无效” 或 “权限不足”。
  • Timeout Errors (超时错误):
    • 如果请求花费的时间超过了服务器设定的响应时间，可能会返回 504 错误，提示网关超时。

4. 特定框架的错误代码

  • 不同的框架如 Django, Spring, Node.js 等，可能会在返回标准 HTTP 错误码的同时，自定义一些错误信息。例如，Django 会将表单验证错误包含在响应体中，Spring 也会使用 @ExceptionHandler 捕获并返回特定的异常错误信息。
